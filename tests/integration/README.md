# Integration Tests

Integration tests are intentionally opt-in because they require live Snowflake
resources. BigQuery/GCS scenarios also need Google Cloud fixtures. S3 Parquet
scenarios need an S3-backed Snowflake stage.

Set `DBT_SNOWFLAKE_ICEBERG_SYNC_RUN_INTEGRATION=1` and provide the environment
variables documented in the root README before running:

```bash
uv run pytest -m integration
```

By default, the integration suite uses the `dbt` executable from the active
Python environment. To run the same tests with dbt Fusion, install the Fusion
CLI and set `DBT_SNOWFLAKE_ICEBERG_SYNC_DBT_EXECUTABLE` to that executable:

```bash
DBT_SNOWFLAKE_ICEBERG_SYNC_DBT_EXECUTABLE=/path/to/dbtf uv run pytest -m integration
```

The BigQuery/GCS suite contains separate tests for:

- concrete non-partitioned BigQuery extract with `auto` and explicit `none`
- BigQuery extract compression with `NONE`, `SNAPPY`, `GZIP`, and `ZSTD`
- native BigQuery `DATETIME` extract mapped to Snowflake `TIMESTAMP_NTZ(6)`
- native BigQuery `TIMESTAMP` extract mapped to Snowflake `TIMESTAMP_NTZ(6)`
  (optional fixture; skipped when `DBT_SNOWFLAKE_ICEBERG_SYNC_BIGQUERY_TIMESTAMP_TABLE_ID` is unset)
- native time-partitioned extract with `auto` and explicit partition decorators
- native integer range-partitioned extract with `auto` and explicit partition decorators
- sharded table extract with all-shard and suffix-filtered plans
- skipped extract behavior when planned BigQuery tables are missing
- query execution export with `auto`, `none`, and `where`
- staging table reuse and forced rebuild behavior
- incremental `delete+copy` over three dbt runs: create, incremental, repeat
- invalid parameter combinations that must fail at the dbt/procedure boundary

The S3 Parquet suite (`test_s3_parquet_sync.py`) is additionally gated on
`DBT_SNOWFLAKE_ICEBERG_SYNC_S3_PARQUET_STAGE` and covers:

- full refresh load from a generated Parquet prefix
- pattern-filtered load
- incremental `delete+copy` with a repeated run (verifies `FORCE = TRUE`)
- empty location skip vs fail
- additive schema evolution
- invalid incremental path/predicate combinations

S3 Parquet fixtures are generated by unloading from Snowflake into the
caller-provided stage under generated prefixes. No AWS credentials are required
in the test environment beyond the Storage Integration attached to that stage.

Mocked procedure-level retry and failed-initial-run cleanup behavior is covered
by unit tests, not the live integration suite.

Fixture BigQuery tables are supplied by environment variables. The tests create
temporary Snowflake procedures, views, Iceberg tables, run logs, BigQuery extract
jobs, GCS files, and S3 Parquet files, but they do not create or delete the
caller-provided BigQuery fixture tables. See the root README for the complete
environment variable list.
